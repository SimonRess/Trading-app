# Changelog

All notable changes to the game are recorded here. This file is the running
record of *what changed and why*; deeper reasoning for design decisions lives in
`docs/decisions/` (ADRs) and system specs in `docs/design/`.

**Keep this in sync:** every change that alters behaviour, data, or
infrastructure gets an entry here, plus an update to the ADR/design doc it
touches. See `docs/00_project_structure.md` §5 (Contribution Workflow).

The format is loosely based on [Keep a Changelog](https://keepachangelog.com).
Dates are `YYYY-MM-DD`.

## [Unreleased]

### Fixed
- **Ship-movement animation never actually played during real turns** — End Turn always routed through a separate `turn-summary` screen branch, which unmounted `<MapView>` (destroying its `MapScene`) on every turn. A freshly-created `MapScene` always starts markers already at their final position, so the glide added earlier only ever worked for the Map↔Port toggle within a turn (which doesn't move ships), never for actual turn-to-turn ship movement. Turn-summary is now an overlay (`.turn-summary-overlay`) rendered on top of the still-mounted port/map view instead of a separate screen, so the map only unmounts on genuine session boundaries (new game, game over).
- **Ship-movement animation finishing invisibly while the map was hidden** — the glide runs on wall-clock time regardless of whether the map is on screen, so a ship that moved while the player was on the Port screen would finish its tween in the background and already be snapped in place by the time they opened the map. `MapScene.setVisible()` now freezes marker positions while hidden and resumes each glide from exactly how far it had progressed before hiding, so switching to the map always shows the motion rather than the already-finished result.
- **Ship pixel-art hull was upside down** — the hull tapered wide-to-narrow going *down* (narrow deck, wide keel), reading as a wedge balanced on its point. Flipped so it's wide at the top (deck) and tapers to a point at the bottom (keel), matching an actual hull silhouette.
- **Map legend not appearing** — `MapScene.mount()` runs while its container is still `display:none` (the "persistent mount" perf fix keeps `<MapView>` always-mounted), so the zero-size guard in `handleResize` skipped positioning the legend's `hudLayer`, leaving it un-positioned unless a `ResizeObserver` firing on the `0×0 → real size` transition happened to correct it — not a reliable cross-browser guarantee. `mount()` now sets an immediate fallback position, and a new `MapScene.refreshLayout()` method is called explicitly by `MapView.svelte` (via a new `visible` prop) whenever `App.svelte` switches to the map screen, so layout no longer depends solely on `ResizeObserver` firing.
- **Ship sprite's sail and hull merging into one shape** — `SHIP_PIXEL_PATTERN` widened straight from the sail's triangular base into the hull's top edge with no separation. Redrawn as a 9-row pattern (was 7) with two single-pixel mast rows between sail and hull, so the icon reads as mast-and-sail atop a hull rather than one blob.

### Changed
- **Winning no longer ends the session** — reaching 10,000+ net worth or Mayor rank still triggers a 'win' outcome (shown once, via a new victory variant of the turn-summary overlay with "Continue Playing →" / "Retire & Play Again" choices), but the player can now choose to keep playing instead of the game forcing a `game-over` screen. `GameState.hasWon` latches permanently the first time a win condition fires, so it's a one-time celebration rather than re-triggering every subsequent turn while net worth/rank remain above threshold. Losing (bankruptcy, out of turns) is unaffected and still ends the session via `game-over`, even after a prior win. Additive save-file field (`hasWon`), no schema bump — older saves default to `false` on load.

### Added
- **Roadmap scoping: ship renaming, church donations, child traits, warehouses (none implemented yet)** — four new/extended feature proposals added to the design-doc set at the player's request, each with a target version in `docs/design/mvp-scope.md`'s out-of-scope table: renaming ships (v1.1, see `ship-stats.md` "Renaming Ships") is small enough to not need its own doc; a per-city church that starts partially built and advances via cash donations, also raising the player's existing `reputation` in that city (v1.1, `docs/design/church-donations.md`); children growing up with optionally-teacher-influenced character traits that carry into the next generation (v2, layered onto `docs/design/family-succession.md`'s "Child Development & Traits" section); and buildable/sellable warehouses generating small flat passive income per turn (v2, `docs/design/warehouses.md`).
- **Family & generational succession, scoped (not yet implemented)** — `PlayerState.age`/`maritalStatus` have existed unused/flavor-only since they were added; `docs/design/family-succession.md` now proposes a first-pass mechanic (age advances yearly, succession triggers at 60, fleet/cash/political rank carry over to a new heir while reputation halves and age/marital status reset) ahead of implementation.
- **Political rank progression** (see `docs/design/political-rank.md`) — `PlayerState.politicalRank`/`reputation` had existed unused since early on; now implemented as a first pass. Selling goods gains +1 reputation in that city (capped 100, never decays yet). Each turn, `evaluateRankUp` promotes the player once both net worth and Lübeck reputation clear a threshold (1,500 Mark/30 rep → Guild Member, 4,000/50 → Council Member, 10,000/75 → Mayor of Lübeck) — never demotes. A rank-up is announced in the turn-summary overlay and shown as a header badge next to name/age/marital status. Reaching Mayor also satisfies the game's win condition (see "Changed" above). Thresholds are a first pass, not yet simulation-tuned.
- **Season order/duration info** — an ⓘ button next to the calendar header opens a note explaining the season cycle (Spring → Summer → Autumn → Winter, 1 turn each, new year after Winter) and the game's total length in years.
- **Player name, age, and marital status in the header** — `PlayerState` gained a `maritalStatus` field (`'single' | 'married' | 'widowed'`, defaults to `'single'`); the header now shows "{name} · Age {age} · {status}" next to the calendar info. Additive save-file field — no schema bump, older saves default to `'single'` on load (`save-system.ts`).
- **Animated ship movement on the map** — ship markers no longer snap straight to their new position when a turn resolves; `MapScene` now keeps a persistent marker per ship and glides it to its new target (port fan-out slot, or a point along its route) over 600ms via a PixiJS `Ticker`, eased with `easeOutCubic`, instead of tearing down and redrawing `shipLayer` on every `update()`. See `docs/design/map-view.md` "Ship Movement Animation".
- **Ship speed shown in the Shipyard** — each ship-type card in the Shipyard now shows its speed relative to the Kogge ("standard speed" / "1.5x slower" / "2x faster", derived from `speedRatio()`) alongside capacity and price.
- **Ship buying UI for all three types** — the Shipyard section now shows one card per ship type (Kogge, Hulk, Schnigge) with capacity/price/description and its own independently-enabled Buy button, instead of a single "Buy Kogge" button. `BUY_SHIP` `GameAction` gained a `shipType` field; `executeBuyShip` takes the chosen type.
- **Hulk and Schnigge ship types** (pulled forward from v1.1, same pattern as save/load and repair/buy earlier) — Hulk: 100 last capacity, 1.5x travel time, 800 Mark. Schnigge: 20 last capacity, 0.5x travel time (floored at 1 turn), 250 Mark. `speedRatio(type)` scales `route.turns` relative to the Kogge baseline rather than reintroducing the earlier "doubled travel time" bug — a Kogge's ratio is exactly 1.0. **Follow-up fix caught before shipping:** the UI's destination-time *preview* (`shipTravelTurns` in `App.svelte`) still showed the Kogge's travel time for other ship types even though the actual game logic was correct (verified via a live-browser check, not just unit tests) — fixed to apply `speedRatio` in the preview too.
- **In-game Save/Load UI** — a "💾 Save" menu (export/import buttons) in the port-screen header, plus an import control on the New Game screen. `GameClient` gained `exportSave()`/`importSave(file)` methods so the UI never reaches into `save-system.ts` directly (CLAUDE.md Hard Rule 2), even though export/import don't fit the `(state, action) => newState` `GameAction` shape.
- **Route-danger colouring on the map** — every route line is now tinted along a calm-to-red gradient reflecting its current combined storm+pirate danger (same `routeDangerFactor` normalisation used for event-pool weighting, ADR-015), with a small fixed legend. Makes the per-route/session risk system (previously invisible backend state) something the player can actually see and plan around.
- **Procedural pixel-art map icons** — cities and ships now render as small pixel-grid sprites (a castle silhouette, a sailboat silhouette) via a new `drawPixelSprite()` helper, instead of smooth vector circles/triangles. `Graphics` fills are vector (always crisp), so a deliberately blocky grid pattern reads as genuine pixel art with no texture/sprite-sheet pipeline needed — closer to ADR-005's long-term direction than the previous placeholder shapes, and the only code that needs to change when real sprite-sheet assets eventually arrive.
- **Per-route & session event risk (ADR-015)** — storm and pirate events are now genuinely route/season-sensitive instead of a flat global roll, and regional danger drifts over a session:
  - New `pirateRisk` table on every route (`routes.ts`), mirroring the existing `stormRisk` table.
  - New `GameState.risk` domain (`RiskState`): a session-persistent multiplier per route and per harvest-eligible city (Danzig), starting at 1.0 and drifting by a small bounded random walk each turn (`driftRiskState` in the new `src/game/systems/risk-system.ts`) — this is the "pirate density in specific areas changes over a session" mechanic.
  - `selectEvent`'s pool weighting now factors in each transiting ship's route risk (normalised against the network-wide average, to avoid distorting the original storm/pirate/harvest balance — an earlier direct-multiplication attempt skewed the mix badly, caught and fixed via a 200-trial simulation before shipping).
  - Storm damage is now **per-ship** (6–22, varying by route risk and durability), not a flat 10 to every ship in transit. Pirate raid targeting is now **weighted by route pirate-risk**, not uniform among ships in transit.
  - `fleet-system.ts`'s `applyStormDamage`/`applyPirateRaid` were refactored to take pre-decided damage/target as parameters rather than rolling their own dice, keeping all event randomness centralised in `event-system.ts`.
  - See ADR-015 for the full design, including why the literal "independent per-leg-per-ship roll" reading of the original `city-graph.md` prose was not implemented as-is.
- **Durability-threshold effects (ship-stats.md, ADR-015)** — fully implemented: a Critical ship (≤25 durability) cannot receive new sail orders (`setDestination` refuses; the port UI shows an explicit warning instead of destination buttons); a Damaged ship (26–50) takes +1 turn to travel; Worn/Damaged ships take extra storm damage. Ship cards now show a durability status label (Seaworthy/Worn/Damaged/Critical), color-coded.
- **Map zoom & pan** — mouse wheel, click-drag, and touch pinch-to-zoom on the map view (1.0×–3.5×), plus double-click/double-tap to reset. Added after feedback that city/ship markers were hard to hit precisely at the default fit-to-screen scale. Documented in `map-view.md`.
- **Mobile responsiveness pass** — below 700px width, the port screen's two-column layout (fixed 220px Fleet sidebar + trade panel) stacks vertically instead of squeezing the trade panel's 6-column market table until its Buy/Sell columns became inaccessible (a real bug caught via Playwright mobile-viewport screenshots, not just theorised). The fold button now collapses the Fleet panel's height (not width) once it's already full-width. The header wraps onto clear separate rows instead of mid-word. Touch targets (buy/sell/destination/shipyard buttons) grow slightly on narrow screens.
- **Map view** — the first real use of the PixiJS game-world layer (ADR-003). A new "🗺️ Map" / "⚓ Port" nav toggle in the port-screen header switches between the existing Port view and a new spatial map: the 5 cities at their documented positions (`cities.ts`), route lines between them (`routes.ts`), and ship markers — in port (fanned out if several ships share a city) or interpolated along their route line while in transit, based on `turnsRemaining / route.turns`. The active route a ship is sailing is highlighted; the selected city gets a ring; the selected ship gets a label and a brighter color. Clicking a city or a ship marker selects it and switches back to the Port view, mirroring the existing city-select buttons and ship cards. Placeholder `Graphics`-primitive art (no pixel-art assets exist yet, per ADR-005). New `src/render/map-scene.ts` (plain PixiJS class, no Svelte/game-system dependency) and `src/ui/MapView.svelte` (thin wrapper). Scales/centers to fit its container via `ResizeObserver`, so it also adapts to the foldable Fleet panel, not just window resizes. Documented in `docs/design/map-view.md`.
- **MVP game systems** — first playable implementation of the core loop:
  - `src/game/data/ships.ts` (Kogge stats, `shipNetWorth`)
  - `src/game/systems/fleet-system.ts` (movement, arrivals, storm damage, pirate raid, cargo helpers)
  - `src/game/systems/event-system.ts` (weighted event selection per `design/event-table.md`)
  - `src/game/systems/turn-system.ts` (full turn resolution, buy/sell, `computeNetWorth`)
  - Playable UI in `src/ui/App.svelte` (new-game → port → turn summary → game over)
- **GitHub Pages deployment** — `.github/workflows/deploy.yml`, `npm run deploy`, and `base` path in `vite.config.ts`. Implements ADR-008. Documented in `docs/design/deployment.md`.
- **Pending sail orders** — choosing a destination in port now sets a *pending* order (changeable/cancellable) that departs only when the turn is ended, instead of departing immediately. UI shows the order on the ship card and in the destination panel.
- **Ship buying & repair (shipyard)** — a "Shipyard" section now appears in the port view whenever the selected ship is docked at Lübeck, Danzig, or Hamburg:
  - **Buy Ship** — 400 Mark for a new Kogge, capped at `MAX_SHIPS` (3) regardless of cash.
  - **Repair** — restores the selected ship to full durability for `(100 - durability) × 2` Mark; disabled at full durability or insufficient cash.
  - New `GameAction`s `BUY_SHIP` and `REPAIR_SHIP`; new pure functions `executeBuyShip`/`executeRepairShip` in `turn-system.ts`; `SHIPYARD_CITIES`, `MAX_SHIPS`, `repairCost`, `nextShipName` added to `src/game/data/ships.ts`.
  - Resolves two open questions in `ship-stats.md`: repair/purchase is restricted to the three shipyard cities (not all five), and the MVP uses a manual shipyard UI rather than auto-charging on port visit.
  - Also documents an existing gap: durability-threshold effects (storm-chance modifiers, travel-time penalty, "cannot depart if critical") are still **not enforced** — see `ship-stats.md` Implementation Status.
- **Destination travel-time preview** — each destination button in the port view now shows the trip length (e.g. "Danzig (2t)"), and the pending-order note states it too ("depart for Danzig (2 turns)..."). Previously the travel time was only visible after committing to a voyage (on the ship card, post-departure). **Follow-up fix:** the first pass missed the ship card itself in the Fleet panel — it showed the pending order ("⚓ → Danzig") without a turn count until the turn was actually ended. The ship card now shows "⚓ → Danzig (2t)" immediately on selecting a destination, matching the destination button and the order note.
- **No-shipyard notice** — the port view now tells the player when the current city has no shipyard (e.g. "Riga has no shipyard. Repairs and new ships are available in Lübeck, Danzig, Hamburg."), instead of the Shipyard section simply not appearing with no explanation.
- **Event message icons** — each turn-summary message now starts with an icon indicating its kind: ⚓ ship arrival, ⛈️ storm, 🌾 bumper harvest, 🏴‍☠️ pirate raid. Defined directly in the message strings (`turn-system.ts`, `event-system.ts`) so any future UI surface gets them for free.
- **Good icons** — each good now has a small icon shown next to its name in the market table: 🧂 Salt, 🌾 Grain, 🪵 Timber, 🦫 Furs, 🐟 Herring. New `src/ui/icons.ts` (`GOOD_ICONS`), presentation-only, not part of `src/game/`.
- **Foldable fleet panel** — the Fleet sidebar can now be collapsed to a narrow strip (a `◀`/`▶` toggle button) to reclaim horizontal space, particularly on small/mobile screens. The panel content re-appears on expand; selection state is preserved while collapsed.

### Changed
- **Net-worth valuation** — held cargo is now valued at each good's fixed base price instead of the fluctuating local market price. Removes the per-turn "paper" drift while idle and closes a hoard-to-win exploit. See **ADR-014**. `mvp-scope.md` and `ship-stats.md` updated to match.
- **Storm damage** corrected to **10 durability** across all docs. `mvp-scope.md` previously said 5, contradicting `ship-stats.md` and `event-table.md` (which say 10, with a note that 5 was too minor). 10 is authoritative.
- **Pirate raid** effect aligned to **15 % of cargo, proportional across goods** in `mvp-scope.md` (previously "10 %"), matching `event-table.md`.
- **ADR-010 (Combat)** moved from Proposed to **Accepted**.

### Fixed
- **Map slow to open the first time** — `App.svelte` used `{#if screen === 'map'}<MapView .../>{/if}`, which tore down and recreated the entire `MapScene` (a new PixiJS `Application`, WebGL context, shader compilation, font textures) on *every* Map/Port toggle, not just the first. "Faster afterwards" was just the browser's GPU driver caching compiled shaders across repeated context creations, not the app reusing anything. Fixed by keeping `<MapView>` permanently mounted and toggling only its container's CSS visibility — see `map-view.md` "Persistent mount".
- **Svelte reactivity in the port UI** — buy/sell, the trade panel during travel, and setting a new destination after arrival did nothing on screen. Root cause: the reactive statement `$: activeShip = shipById(selectedShipId)` only tracked `selectedShipId`; Svelte cannot see `state` reads inside a called function, so with a single ship (whose id never changes) the derived ship/port never recomputed. Fixed by referencing `state.fleet.ships` directly in the reactive statement.
- **CI lint** — resolved 32 ESLint errors (import boundaries, `no-default-export`, `require-await`, non-null assertions in tests, template-literal number coercion, dynamic delete) so the deploy pipeline is green.
- **Travel time silently doubled** — `fleet-system.ts`'s `setDestination` computed `turnsRemaining` as `route.turns × SHIP_TYPES[type].turnsPerLeg`, but `route.turns` (per `city-graph.md`) already *is* the full travel time "assuming a Kogge at standard speed" — the multiplication double-counted the Kogge's speed on every voyage (e.g. Malmö→Riga took 6 turns in-game instead of the documented 3). This was reported as "no storms in 40 turns while sailing Malmö↔Riga": the underlying storm probability was verified correct by simulation (~3.9 storms expected per 40-turn game, 0/200 zero-storm trials), but the doubled travel time roughly halved the number of voyages actually completed, sharply cutting the number of storm rolls a player would see. Fixed by using `route.turns` directly; see `city-graph.md` Implementation Status.

## How to add an entry

When you make a change, add it under `## [Unreleased]` in the right group:

- **Added** — new features, files, or capabilities
- **Changed** — behaviour/data changes to existing features
- **Fixed** — bug fixes
- **Removed** — deleted features

If the change involves a design decision (a real trade-off with alternatives),
write or update an ADR and link it. If it changes how a system works, update the
matching `docs/design/*.md`. The changelog entry should be short; the ADR/design
doc holds the detail.
